Relay supports bridging from Bitcoin to any supported destination chain. This
enables users to move BTC into stablecoins, ETH, or other assets on chains like
Base, Optimism, Arbitrum, and beyond. Bitcoin bridging operates exclusively
through [deposit addresses](/features/deposit-addresses), which allow users to
send BTC to a designated address without requiring wallet connection or
transaction signing on the origin chain.

<Note>
	Currently, only deposits **from** Bitcoin are supported. Withdrawals to
	Bitcoin are not yet available.
</Note>

## How It Works

Bitcoin bridging leverages Relay's deposit address infrastructure. The flow
differs from EVM-to-EVM bridges in several important ways:

1. **Request a quote** with the Bitcoin origin chain and your desired
   destination
2. **Receive a deposit address** — a unique Bitcoin address generated for your
   transaction
3. **Send BTC** to the deposit address from any Bitcoin wallet
4. **Relay detects the deposit** after 1 block confirmation
5. **Relay sweeps the funds** from the deposit address to the solver wallet
6. **The solver fills** the order on the destination chain

Because Bitcoin lacks smart contract capability, there is no calldata or
on-chain signing involved on the origin side. You simply transfer BTC to the
provided address.

<Info>
	The current flow requires waiting for both the deposit transaction and
	the sweep transaction to confirm before the fill executes. Relay is
	actively working on an improvement to fill immediately after the deposit
	confirms, which will significantly reduce end-to-end bridging time.
</Info>

## Block Confirmations and Timing

Bitcoin's block time averages approximately 10 minutes, though this varies
considerably in practice. Blocks can arrive within minutes of each other, or the
network may go 30-50 minutes between blocks depending on mining variance.

Relay waits for **1 block confirmation** before processing a deposit. Once
confirmed:

- Relay attempts to sweep the funds to the solver wallet in the next block
- This sweep is **best effort and not guaranteed** to occur in the immediate
  next block
- The solver then fills the order on the destination chain

<Warning>
	Due to Bitcoin's variable block times and the multi-step nature of the
	flow, do not set aggressive timeouts in your polling logic. Allow for at
	least 2-3 blocks worth of time (potentially 20-40 minutes in adverse
	conditions) before considering a transaction stalled.
</Warning>

You can monitor Bitcoin block times and mempool congestion at
[mempool.space](https://mempool.space).

<Note>
	For high-value requests, additional confirmation requirements may be
	introduced in the future to mitigate reorg risk.
</Note>

## Quote Requirements

Bitcoin quotes have specific requirements that differ from EVM chains.

### User Address

Unlike EVM chains where you can pass the zero address
(`0x0000000000000000000000000000000000000000`) for indicative quotes, Bitcoin
has no equivalent null address. You must provide a **valid Bitcoin address** in
the `user` parameter.

### Balance Requirement

The address provided in the `user` field must hold a balance that **exceeds the
quoted amount**. If the balance is insufficient, the API returns an
`INSUFFICIENT_FUNDS` error.

<Tip>
	For indicative quotes or UI previews where the actual sender is unknown,
	use a known high-balance address (sometimes called a "whale address").
	This allows you to fetch quotes without requiring end-user wallet
	connection. The actual deposit can originate from any address — the
	quote constraint applies only to the `user` parameter at quote time.
</Tip>

This limitation is temporary. Relay plans to decouple deposit address quotes
from user balance requirements in a future update.

## API Parameters

Bitcoin can be accessed using the standard Relay API flow. Review the
[execution steps](/references/api/api_core_concepts/step-execution)
documentation to understand the general pattern, then use the
[Get Quote](/references/api/get-quote-v2) endpoint.

### Bitcoin-Specific Parameters

| **Action**           | **Parameter**       | **Input**                                   | **Description**                                                         |
| -------------------- | ------------------- | ------------------------------------------- | ----------------------------------------------------------------------- |
| Deposit from Bitcoin | `originChainId`     | `8253038`                                   | Chain ID assigned to Bitcoin in Relay                                   |
|                      | `user`              | Valid Bitcoin address                       | Must be a valid Bitcoin address with balance exceeding the quote amount |
|                      | `originCurrency`    | `bc1qqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqmql8k8` | Native BTC identifier (Bitcoin's equivalent of a null address)          |
|                      | `useDepositAddress` | `true`                                      | Required for Bitcoin — bridging only works via deposit addresses        |

All other standard API parameters (`destinationChainId`, `destinationCurrency`,
`recipient`, `amount`, `tradeType`) function as documented.

## Example Request

The following example bridges 0.001 BTC from Bitcoin to USDC on Base:

```bash
curl -X POST 'https://api.relay.link/quote/v2' \
  -H 'Content-Type: application/json' \
  -d '{
    "user": "bc1q4vxn43l44h30nkluqfxd9eckf45vr2awz38lwa",
    "originChainId": 8253038,
    "originCurrency": "bc1qqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqmql8k8",
    "destinationChainId": 8453,
    "destinationCurrency": "0x833589fcd6edb6e08f4c7c32d4f71b54bda02913",
    "recipient": "0x03508bb71268bba25ecacc8f620e01866650532c",
    "tradeType": "EXACT_INPUT",
    "amount": "100000",
    "useDepositAddress": true,
    "refundTo": "bc1q4vxn43l44h30nkluqfxd9eckf45vr2awz38lwa"
  }'
```

**Notes:**

- `amount` is specified in satoshis (100000 satoshis = 0.001 BTC)
- `refundTo` should be a valid Bitcoin address to receive refunds if the bridge
  fails
- The `user` address must hold sufficient BTC balance for the quote to succeed
- Replace `recipient` with the destination EVM address that should receive funds

The response includes a `depositAddress` field containing the Bitcoin address
where funds should be sent, along with a `requestId` for tracking.

## SDK Integration

To use the SDK for Bitcoin bridging, install and configure the Bitcoin wallet
adapter:

```bash
npm install @relayprotocol/relay-bitcoin-wallet-adapter
```

The adapter uses [bitcoinjs-lib](https://www.npmjs.com/package/bitcoinjs-lib)
under the hood for transaction signing and submission.

```typescript
import { getClient } from "@relayprotocol/relay-sdk";
import { adaptBitcoinWallet } from "@relayprotocol/relay-bitcoin-wallet-adapter";

// Create the Bitcoin wallet adapter
const adaptedWallet = adaptBitcoinWallet(
	walletAddress, // Your Bitcoin wallet address (string)
	async (address, psbt, dynamicParams) => {
		try {
			// Sign using your wallet provider (Dynamic, Leather, Xverse, etc.)
			const response = await primaryWallet.signPsbt(
				dynamicParams
			);
			if (!response) {
				throw "Missing psbt response";
			}
			return response.signedPsbt; // Return signed PSBT as base64 string
		} catch (e) {
			throw e;
		}
	},
	publicKey // Optional third param: wallet's public key
);

// Get a quote using SDK parameter names
const quote = await getClient().actions.getQuote({
	chainId: 8253038, // Bitcoin chain ID
	toChainId: 8453, // Base chain ID
	currency: "bc1qqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqmql8k8", // Native BTC
	toCurrency: "0x833589fcd6edb6e08f4c7c32d4f71b54bda02913", // USDC on Base
	user: walletAddress,
	recipient: "0x03508bb71268bba25ecacc8f620e01866650532c",
	tradeType: "EXACT_INPUT",
	amount: "100000", // in satoshis
	options: {
		useDepositAddress: true,
		refundTo: walletAddress,
	},
});

// Execute the quote
await getClient().actions.execute({
	quote,
	wallet: adaptedWallet,
	onProgress: ({
		steps,
		currentStep,
		currentStepItem,
		txHashes,
		details,
	}) => {
		console.log(`Step: ${currentStep?.action}`);
		console.log("Transaction hashes:", txHashes);
	},
});
```

<Info>
	Bitcoin transactions have extended finalization periods compared to EVM
	chains. The SDK does not poll Bitcoin confirmations in real-time. Build
	your UI to accommodate potential delays of 10-30+ minutes for the origin
	transaction to confirm.
</Info>

## Tracking and Debugging

Use the `requestId` returned in the quote response to track bridge status via
the [Get Execution Status](/references/api/get-intents-status-v3) endpoint. The
Bitcoin transaction hash alone cannot be used to look up bridge status — always
persist and use the `requestId`.

```bash
curl 'https://api.relay.link/intents/status?requestId=YOUR_REQUEST_ID'
```

## Current Limitations

- **Deposits only**: Withdrawals from EVM chains to Bitcoin are not currently
  supported
- **Balance verification**: The quoted `user` address must hold sufficient BTC
  balance
- **No calldata execution**: Destination chain execution is limited to token
  transfers; arbitrary contract calls on destination are not supported for
  Bitcoin-origin bridges
- **Confirmation variability**: Bitcoin block times are unpredictable; plan for
  delays in your UX
- **Two-step confirmation**: Currently, both the deposit and sweep transactions
  must confirm before the fill executes (improvement in progress)
