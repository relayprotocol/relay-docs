  <Step title="Execute">
    The [quote endpoint](/references/api/get-quote-v2) returns a steps array. Think of this as a recipe your application must follow. You need to iterate through these steps and prompt the user to sign or submit them.
    <Callout icon="person-swimming" color="#4615C8">
      ### Deep Dive
      For the full logic on parsing steps, see [Understanding Step Execution](/references/api/api_core_concepts/step-execution).

    ## The Logic
    For a simple ETH bridge, the `steps` array contains a single transaction item. To execute it:

    - **Check the Step Kind:** Identify if the step is a `transaction` (submit to chain) or a `signature` (sign off-chain).
    - **Execute:** For transactions, submit the provided `data`, `to`, `value`, and `chainId` using the user's wallet. For signatures, follow the `signatureKind` provided in the step item to sign the `message`.

    <Callout icon="plus" color="#4615C8">
    ### Need More Complexity?
    For the full logic on parsing steps, see [Understanding Step Execution](/references/api/api_core_concepts/step-execution). Because Relay's API is generic, this exact same loop automatically handles advanced flows:

    - [**Cross-Chain Swaps**](/references/api/api_guides/bridging-integration-guide): Uses the same logic as bridging; just change the input/output currencies in the Quote.
    - [**Smart Accounts**](/references/api/api_guides/smart_accounts/smart-accounts): Batches multiple steps into a single atomic user operation.
    - [**Contract Calls**](/references/api/api_guides/calling-integration-guide): Executes arbitrary logic (like NFT mints) on the destination chain.
    </Callout>

    ## Status Lifecycle
    The diagram below illustrates the journey of a cross-chain transaction through Relay's infrastructure and the corresponding API statuses. Below are descriptions for each status. Refer to the [Get Status](/references/api/get-intents-status-v3) endpoint for a full list of statuses and what they mean.

    - `waiting`: The user is submitting or has submitted the deposit transaction to the Relay Depository contract on the origin chain, but it has not yet been indexed by the system.
    - `pending`: The deposit was successfully indexed. The Relay Solver is now monitoring the request and preparing the fill transaction on the destination chain.
    - `success`: The Relay Solver successfully executed the "Fill Tx", and the funds have reached the recipient.
    
    ![Status Lifecycle](/images/status-lifecycle.png)

    The diagram also depicts failure paths where a transaction may end up in a failure or refund state if the fill cannot be completed. Learn more about how Relay helps you handle [refunds](https://docs.relay.link/references/api/api_core_concepts/refunds).
