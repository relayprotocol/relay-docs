## Introduction

The Relay Protocol is a cross-chain payments system that connects users to solvers (also called relayers) willing to perform onchain actions on their behalf. Relay's architecture separates the concerns of deposit, market making, liquidity providing, and intent filling, enabling significant cost and speed advantages over traditional bridges. The protocol is designed to be open and trustless, making it particularly well-suited for low-value, high-frequency cross-chain actions common in a world of chain abstraction.

You can interact with the Relay Protocol through the [Relay App](https://relay.link), and find the source code for all components on [GitHub](https://github.com/relayprotocol/).

## The Problem: Cross-Chain Bridging Costs

Traditional bridges require users to lock funds on one chain and wait for a message or proof to be verified on another chain before receiving funds. This is slow (minutes to hours) and expensive (high gas costs for verification).

Intent-based protocols like Across improved on this by introducing **relayers** (also called **solvers**) who front their own capital to fill user requests immediately:

1. User **escrows** funds on the origin chain
2. Relayer **fills** the order on the destination chain using their own capital
3. Relayer later **settles** on the origin chain to claim the user's escrowed funds as payment

![Cross-chain relaying flow](/images/flow1.png)

This design provides two key advantages:

- **Speed** - Relayers front their own capital and fill optimistically without waiting for global consensus
- **Cost** - Rebalancing happens infrequently and in batches, amortizing the security cost of bridge usage across many users

However, significant costs remain in the **deposit** (~77,000 gas) and **fill** (~120,000 gas) steps. The Relay Protocol targets these remaining costs with a novel architecture.

## Protocol Architecture

The Relay Protocol separates cross-chain payment operations into four independent but composable components. Each can work standalone or together, and each is open-source.

### Depository

The [Depository](./depository/) is the user's entry point into the protocol. It consists of smart contracts deployed across supported chains (both EVM and Solana) that accept user deposits of native tokens and ERC-20/SPL tokens. Each deposit can carry an optional order ID that links it to a specific cross-chain intent.

From the protocol's perspective, depositing funds into a Depository contract represents a deposit into the Relay Hub. The Depository also handles withdrawals via allocator-signed messages.

- Supports EVM chains (native + ERC-20) and Solana (native + SPL tokens)
- Open-source: [GitHub](https://github.com/relayprotocol/relay-protocol-contracts)

### Oracle

The [Oracle](./oracle/) serves as the communication channel between external chains and the Relay Hub. Since the Hub has no direct access to external chain state, the Oracle attests to key events:

- **Depository deposits** - Verifies that a user deposited funds on an external chain
- **Depository withdrawals** - Tracks whether allocator withdrawal signatures have been executed, are pending, or have expired
- **Solver fills** - Confirms that a solver successfully filled a user's order on the destination chain
- **Solver refunds** - Confirms that a solver refunded an order it could not fill

Reference implementation: [GitHub](https://github.com/relayprotocol/relay-protocol-oracle)

### Hub

The [Hub](./hub/) is the core component that tracks ownership of funds across the protocol. It processes oracle attestations of external actions and updates its internal state accordingly. The Hub also supports direct user actions:

- **Withdrawals** - Users request withdrawal of available funds, which are held in a lock until an oracle attestation confirms the withdrawal on the destination chain
- **Unlocks** - If a solver neither fills nor refunds an order before expiry, anyone can trigger an unlock to release the order-locked balance

The current Hub implementation is a centralized service, with plans to evolve toward a decentralized design (e.g., the Hub as its own chain).

Reference implementation: [GitHub](https://github.com/relayprotocol/relay-protocol-hub)

### Vaults

[Vaults](./vaults/) are permissionless, ERC-4626 compliant, yield-bearing liquidity pools that solvers use to rebalance capital across chains. Depositors earn a **base yield** from lending protocols (e.g., Aave) while their funds are idle, plus a **boost yield** when solvers utilize the funds for cross-chain rebalancing.

Vaults solve the solver capital efficiency problem: instead of locking up capital on every supported chain, solvers can tap into shared liquidity pools for instant rebalancing.

- Deployed on Ethereum Mainnet and Arbitrum
- Open-source: [GitHub](https://github.com/relayprotocol/relay-vaults)

## End-to-End Flow

A typical cross-chain transaction through the Relay Protocol:

1. **Deposit** - Alice sends funds to the [Depository](./depository/) contract on the origin chain, with an order ID attached
2. **Fill** - The [solver](/how-it-works/the-relay-solver) detects the deposit and fills Alice's order on the destination chain using its own capital
3. **Track** - The [Hub](./hub/) records the deposit and associates it with the order ID
4. **Verify** - The solver triggers the [Oracle](./oracle/) to verify the fill was completed correctly
5. **Settle** - Once verified, the solver claims the deposited funds from the Hub as payment
6. **Rebalance** - The solver replenishes its capital on destination chains using [Vaults](./vaults/), which batch rebalancing through slow bridges for cost efficiency

## Security

The protocol is developed and deployed incrementally, with each component audited by leading security firms. See the [Audits](./audits) page for a full list of completed security reviews.

## Current State and Roadmap

The protocol components listed above are implemented and in production. They continue to evolve toward a fully trustless cross-chain payments system:

- **Today** - The Relay Solver operates as a trusted relayer, minimizing gas costs to approximately 42,000 gas total (origin transfer + destination transfer)
- **In progress** - The Depository, Oracle, Hub, and Vaults provide the building blocks for decentralized settlement, removing the need for trusted intermediaries
- **Future** - Full decentralization of the solver network and Hub, eliminating any single point of trust

<Info>
These components will evolve over time. The current implementations provide an intermediate state that enables Relay to progress from a simple intent-filling API to a fully trustless cross-chain payments system.
</Info>
